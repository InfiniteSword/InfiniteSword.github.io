JS 内存泄露

js内存泄露是指一块被分配的内存在浏览器进程结束前既不能使用，又不能回收的问题。在js中，内存是根据对象的创建分配给该对象的，并会在没有对该对象的引用时由浏览器收回。这种机制本身没有问题，问题在于浏览器在为 DOM 对象分配内存和恢复内存的方式。

Internet Explorer 和 Mozilla Firefox 均使用引用计数来为 DOM 对象处理内存。在引用计数系统，每个所引用的对象都会保留一个计数，以获悉有多少对象正在引用它。如果计数为零，该对象就会被销毁，其占用的内存也会返回。

循环引用

当两个对象互相引用时，就构成了循环引用，其中每个对象的引用计数值都被赋 1。在纯垃圾收集系统中，循环引用问题不大，若涉及到的两个对象中的一个对象被任何其他对象引用，那么这两个对象都将被垃圾收集。而在引用计数系统，这两 个对象都不能被销毁，原因是引用计数永远不能为零。在同时使用了垃圾收集和引用计数的混合系统中，将会发生泄漏，因为系统不能正确识别循环引用。在这种情况下，DOM 对象和 JavaScript 对象均不能被销毁。

闭包导致的循环引用

闭包使内部变量在外部变量返回时也能保留对外部函数变量的引用。

闭包非常易于隐藏 JavaScript 对象和 DOM 对象间的循环引用。如果一个构成闭包的函数对象被指定给了一个 DOM 节点的事件处理器，而对该节点的引用又被指定给函数对象作用域中的一个活动（或可变）对象，那么就存在一个循环引用。

解决方法：

1、解除闭包关系，把事件处理函数定义到bindEvent()外边。

2、或者在定义事件处理函数的外部函数中，删除对dom的引用（obj.onclick后面注释掉的一行）。

解决方法：将e.innerHTML改为this.innerHTML就可以避免。

页面交叉泄漏

页面交叉泄漏其实是一种较小的泄漏,这种基于插入顺序而常常引起的泄漏问题，主要是由于对象创建过程中的临时对象未能被及时清理和释放造成的。它一般在动态创建页面元素，并将其添加到页面DOM中时发生。

一个最简单的示例场景是我们动态创建两个对象，并创建一个子元素和父元素间的临时域。然后，当你将这两个父子结构元素构成的的树添加到页面DOM树中时，这两个元素将会继承页面DOM中的层次管理域对象，并泄漏之前创建的那个临时域对象。

两种动态创建并添加元素到页面DOM中的方法：

1、将每个子元素添加到它的直接父元素中，最后再将创建好的整棵子树添加到页面DOM中。当一些相关条件合适时，这种方法将会由于临时对象问题引起泄漏。

2、自顶向下创建动态元素，并使它们被创建后立即加入到页面DOM结构中去。由于每个被加入的元素继承了页面DOM中的结构域对象，我们不需要创建任何的临时域。这是避免潜在内存泄漏发生的好方法。

貌似泄漏

在大多数时候，一些APIs的实际的行为和它们预期的行为可能会导致你错误的判断内存泄漏。貌似泄漏大多数时候总是出现在同一个页面的动态脚本操作中，而在从一个页面跳转到空白页面的时候发生是非常少见的。

和DOM插入顺序问题那样，这个问题也需要依赖创建临时对象来产生”泄漏”。对一个脚本元素对象内部的脚本文本一而再再而三的反复重写，慢慢地你将开始泄漏各种已关联到被覆盖内容中的脚本引擎对象。特别地，和脚本调试有关的对象被作为完全的代码对象形式保留了下来。

如果你运行上面的示例代码并使用任务管理器查看，当从”泄漏”页面跳转到空白页面时，你并不会注意到任何脚本泄漏。因为这种脚本泄漏完全发生在页面内部，而且当你离开该页面时被使用的内存就会回收。对于我们原本所期望的行为来说这样的情况是糟糕的。你希望当重写了脚本内容后，原来的脚本对象就应该彻底的从页面中消失。但事实上，由于被覆盖的脚本对象可能已用作事件处理函数，并且还可能有一些未被清除的引用计数。正如你所看到的，这就是貌似泄漏。在表面上内存消耗量可能看起来非常的糟糕，但是这个原因是完全可以接受的。

CSS动画优化

最近写了个自定义样式的audio播放器，在设置外观的时候遇到了严重的卡顿问题，于是整理了下css的动画优化。

部分内容收集自前端性能优化（CSS动画篇），高性能 CSS3 动画。

关于页面绘制

原理

现代浏览器在使用CSS3动画时，以下四种情形绘制的效率较高，分别是：

  • 改变位置
  • 改变大小
  • 旋转
  • 改变透明度

浏览器在渲染一个页面时会经过6个步骤：

  • 获取DOM后分割为多个图层
  • 对每个图层的节点计算样式结果（Recalculate style–样式重计算）
  • 为每个节点生成图形和位置（Layout–回流和重布局）
  • 将每个节点绘制填充到图层位图中（Paint Setup和Paint–重绘）
  • 图层作为纹理上传至GPU
  • 符合多个图层到页面上生成最终屏幕图像（Composite Layers–图层重组）

Chrome中满足以下任意情况就会创建图层：

  • 3D或透视变换（perspective transform）CSS属性
  • 使用加速视频解码的<video>节点
  • 拥有3D（WebGL）上下文或加速的2D上下文的<canvas>节点
  • 混合插件（如Flash）
  • 对自己的opacity做CSS动画或使用一个动画webkit变换的元素
  • 拥有加速CSS过滤器的元素
  • 元素有一个包含复合层的后代节点（一个元素拥有一个子元素，该子元素在自己的层里）
  • 元素有一个z-index较低且包含一个复合层的兄弟元素（换句话说就是该元素在复合层上面渲染）

如果图层中某个元素需要重绘，那么整个图层都需要重绘。比如一个图层包含很多节点，其中有个gif图，gif图的每一帧，都会重绘整个图层的其他节点，然后生成最终的图层位图。所以这需要通过特殊的方式来强制gif图属于自己一个图层（translateZ(0)或者translate3d(0,0,0)），CSS3的动画也是一样。（好在绝大部分情况浏览器自己会为CSS3动画的节点创建图层）

层和CSS动画

每一帧动画浏览器可能需要做如下工作：

  • 计算需要被加载到节点上的样式结果（Recalculate style–样式重计算）
  • 为每个节点生成图形和位置（Layout–回流和重布局）
  • 将每个节点填充到图层中（Paint Setup和Paint–重绘）
  • 组合图层到页面上（Composite Layers–图层重组）

如果我们需要使得动画的性能提高，需要做的就是减少浏览器在动画运行时所需要做的工作。最好的情况是，改变的属性仅仅影响图层的组合，变换（transform）和透明度（opacity）就属于这种情况。

现代浏览器如Chrome，Firefox，Safari和Opera都对变换和透明度采用硬件加速，但IE10+不是很确定是否硬件加速。

别使用CSS类名做状态标记

如果在网页中使用CSS的类来对节点做状态标记，当这些节点的状态标记类修改时，将会触发节点的重绘和重布局。所以在节点上使用CSS类来做状态标记代价会很大。【很多时候为了方便用这种方法，实际上却是很不可取的】

重布局与重绘

有些属性，当你改变他时，会需要对节点重新布局（这也意味着需要重新计算其他被影响的节点的位置和大小）。这种情况下，被影响的DOM树越大（可见节点），重绘所需要的时间就会越长，而渲染一帧动画的时间也相应变长。

有些属性不会修改节点的大小和位置，不会触发重布局，但是节点内部的渲染效果进行了改变，所以只需要重绘就可以了。

在重绘时，这些节点会被加载到GPU中进行重绘，这对移动设备如手机的影响还是很大的。因为CPU不如台式机或笔记本电脑，所以绘画巫妖的时间更长。而且CPU与GPU之间的有较大的带宽限制，所以纹理的上传需要一定时间。所以手机即使重绘也会很慢。

图层重组

  • 透明度【透明度并不触发重绘，因为透明度的改变仅仅是GPU简单降低了之前画好的纹理的alpha值达到的效果，当然，如果该图层之下还有图层，他们也会被透明化】
  • 使用translateZ(0)或者translate3d(0,0,0)使浏览器创建图层，可以消除在动画开始之前的图层创建时间，使得动画尽快开始（创建图层和绘制图层还是比较慢的），而且不会随着抗锯齿发生突变。不过这种方法需要节制，否则会因为创建过多的图层导致崩溃。
  • Chrome中的抗锯齿，非根图层以及透明图层使用grayscale antialiasing而不是subpixel antialiasing，如果抗锯齿方法变化，这个效果将会非常显著。如果你打算预处理一个节点而不打算等到动画开始，可以通过这种强迫浏览器创建图层的方式进行。
  • 我们通过节点的transform可以修改节点的位置、旋转、大小等。我们平常会使用left和top属性来修改节点的位置，但正如上面所述，left和top会触发重布局，修改时的代价相当大。取而代之的更好方法是使用translate，这个不会触发重布局。【PC下，这些区别并不明显，而移动端这些优势就能体现出来】

关于移动端

相比PC端，移动端的css动画会出现很多性能问题，主要体现在卡顿与闪烁，而且移动端用js做动画也并不是一个很好的方案。那么要如何提升移动端css性能呢？

  • 使用translate变换比修改left等值的动画流畅度明显更高。

  • 使用transform: translate3d(0, 0, 0);能开启GPU加速，但是会增加内存使用。

XSS相关

跨站脚本(Cross-site Scripting)即XSS，为不和层叠样式表(Cascading Style Sheets)的缩写混淆，故缩写为XSS。指攻击者向目标Web站点注入HTML标签或者脚本这类安全问题。

在之前身边发生了某个关于注入的事件，记忆深刻…所以写下这篇来提醒自己。

关于XSS过滤

说到XSS过滤，我们应该想到的是有多少种绕过过滤的方法。过滤<script>标签可以防止部分js代码的注入，但绝对并不周全。除了直接插入js代码，还有通过事件处理器插入js代码的方法，比如<body onload="xxx">以及<img src="" onerror="xxx">等。对此，作为前端的学习者，避免使用内联的事件处理是必要的。

有一个用js写的XSS攻击代码过滤模块，github地址: https://github.com/leizongmin/js-xss，可以通过 npm install xss 来安装。

###防止XSS攻击

1 防止XSS攻击，通常是在使用不可信数据动态创建文档内容前移除HTML标签。如将<script>等标签的尖括号替换为它们对应的HTML实体。这种方法可归于输入过滤。通过html标签注入是较常见XSS注入攻击，入口常为表单元素以及内联样式表。通常表单项会经过正则匹配对提交内容进行过滤，PHP中可以使用htmlspecialchars或者htmlentities函数进行转义。

2 对用户提交的数据进行重编码。在用户提交表单之后，将数据通过HtmlEncoder等工具编码后再输出到目标页面。重编码之后，攻击者输入的代码会被当做普通的文字，这样来防止浏览器执行代码。

3 IE8中定义了一个toStaticHTML()方法，用于移除<script>标签以及其他潜在可执行内容而不修改不可执行的HTML。HTML5为<iframe>元素定义了一个sandbox属性。实现之后允许显示不可信的内容，并自动禁用脚本。

4 防止攻击者获取cookie信息。避免在cookie中出现用户名、密码等隐私信息，将cookie通过算法进行多次散列再存放，以及将cookie与id进行绑定。其实在实际项目中，更多的是后台使用session来存储重要信息，而为了安全cookie只用来辅助存储非重要信息。

5 在表单中添加hash值，在thinkphp中称为表单令牌的验证，这种处理除了防止表单的重复提交，也能防止从该页面之外的地方提交表单，即从某种方面防止了跨站攻击。

6 使用验证码。

7 避免漏洞 _(:з」∠)_

关于placeholder的兼容实现

今天写一个表单样式的时候遇到了js事件监听函数的兼容性问题，记录下来。

由于input标签的placeholder属性在早期的ie浏览器是没有的，于是我采用js修改input标签的value值来模拟placeholder的效果。

我的思路是先判断浏览器是否支持addEventListener,如果支持，则使用它，而且火狐和谷歌浏览器对placeholder属性都能很好地支持，不支持addEventListener就使用attachEvent来添加事件，同时用js模拟placeholder效果。这样理应是没问题的，但是在ie9下的input里始终是空白的，ie10和ie8都会出现内容。

反复检查代码之后，想到了一种可能性，于是把判断改成了先判断浏览器是否支持attachEvent，不支持则用另一个。果然，ie9下成功出现了placeholder的效果。

如我想的一样，ie9已经开始支持addEventListener的事件监听，但是却并不支持placeholder属性，在我之前的思路里，它根本没有经过我模拟placeholder的代码。
